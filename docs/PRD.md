# WalkMate — PRD (기획서 / 기능 명세)

> 가칭 **WalkMate** (레포·패키지명은 `netzero` 유지). 반려견 보호자를 위한 통합 앱.
> 본 문서는 해커톤 MVP **백엔드 범위**의 기능 명세이며, 사용자 흐름과 수용 기준(Acceptance
> Criteria)을 중심으로 기술한다. 상세 인터페이스는 [api-spec](./api-spec.md),
> 데이터 모델은 [domain-model](./domain-model.md), LLM은 [llm-spec](./llm-spec.md),
> 배포는 [infra](./infra.md) 참조.

## 0. 한 줄 요약 / 구성 축

반려견 보호자가 **근처 산책 메이트를 찾고(매칭)**, **애견용품을 거래하고(중고/저탄소 마켓)**,
소통하는 통합 앱. 세 축으로 구성된다.

| 축 | 설명 | 백엔드 범위 |
| --- | --- | --- |
| 1. 산책 매칭 (**핵심**) | GPS 기반 근처 메이트 탐색 · 친구 · 1:1 채팅 | ✅ 포함 |
| 2. 중고거래 / 저탄소 마켓 | 애견용품 중고거래 + 저탄소 큐레이션 마켓 | ✅ 포함 |
| 3. 커뮤니티 | 게시판/피드 | ❌ **Out of Scope** (프론트 화면만, 백엔드 작업 없음) |

## 2. 온보딩 / 사용자

### 2.1 흐름
1. 간단 가입(email/password) → JWT 발급 (Q7).
2. 온보딩 입력: **닉네임**, **성별**, **강아지와 함께 나오는지 여부**(`hasDog` boolean).
3. `hasDog == true` 인 경우에만 강아지 정보 입력: **강아지명, 성별, 품종, 나이(선택), 성향**.
   `false` 면 강아지 정보 없음.
4. 메인(지도) 진입.

### 2.2 수용 기준
- [ ] 회원가입/로그인 시 JWT가 발급되고, 인증 필요한 API는 토큰 없으면 401.
- [ ] `hasDog == false` 이면 강아지 정보 입력/저장이 요구되지 않는다.
- [ ] `hasDog == true` 이면 강아지명·성별·품종·성향은 필수, 나이는 선택.
- [ ] 본인 정보 전체 조회(`GET /api/users/me`)가 **사용자 + 강아지 정보**를 함께 반환한다.
- [ ] (가정) 강아지는 1마리. 데이터 모델은 1:N 확장이 가능하도록 설계한다.

## 3. 산책 매칭 (최우선 핵심)

### 3.1 실시간 위치 + 고스트모드
**흐름**: 클라이언트가 GPS 좌표를 주기적으로 서버에 PATCH → 지도에 본인 + 근처 사람 표시.

- 위치 갱신: `PATCH /api/users/me/location` `{lat,lng}` (위치 + `last_active_at` 갱신). (Q9 폴링)
- **고스트모드**(`ghostMode` 토글): ON이면 본인 위치가 지도/목록에 **노출되지 않음**, OFF면 노출.

**수용 기준**
- [ ] 위치 PATCH 시 `last_active_at`이 갱신된다.
- [ ] 고스트모드 ON 사용자는 근처 사람 목록과 지도 실시간 핀 양쪽에서 제외된다.
- [ ] 본인 좌표는 본인에게는 항상 보인다(고스트 여부 무관).

### 3.2 근처 사람 목록
**흐름**: 지도 상단 "근처 사람" 버튼 → 반경 내 사용자 목록 → 항목 클릭 → 상세 → "메시지 보내기" → 대화창.

- 항목 필드: **프로필 이미지**(사용자/강아지 첫 이미지), **마지막 접속 시간**, **닉네임**.
- 필터: **반경 2km 이내**(Q5) + **`last_active_at` 24시간 이내**(Q6) + **비고스트** + **본인 제외**.

**수용 기준**
- [ ] 24시간 지난 사용자는 목록에서 제외된다.
- [ ] 거리 계산은 Haversine 쿼리로 반경 내만 반환한다.
- [ ] 목록 항목의 프로필을 통해 채팅(메시지 보내기)으로 진입할 수 있다.

### 3.3 지도 위 실시간 프로필
- 지도에 표시된 실시간 위치 핀 클릭 → 해당 사용자 프로필 표시 → **거기서도 채팅 시작 가능**.
- 실시간 핀: online(최근 N분) **비고스트** 사용자 (`GET /api/walk/map-users`).

### 3.4 친구 기능
**흐름**: 친구 요청 → 상대 수락 → 친구 성립.

- 본인이 **보낸** 친구 요청 목록 / 본인이 **받은** 친구 요청 목록 조회.
- 지도 화면 위 친구 목록: **접속 중(online) 친구가 왼쪽부터** 정렬, 접속 여부를 **초록 원**(=`online` 필드)으로 표시.
- 친구 프로필 상세 조회.

**수용 기준**
- [ ] 요청 → 수락으로만 친구가 성립한다(자동 친구 없음).
- [ ] 친구 목록 응답에 `online` boolean이 포함되고, online 친구가 먼저 정렬된다.
- [ ] 보낸/받은 요청 목록을 각각 조회할 수 있다.

### 3.5 채팅
**흐름**: 대화창에서 1:1 메시지 송수신, 메시지는 **영속화**.

- 대화방(스레드) 목록 / 특정 대화방 메시지 조회 / 메시지 전송.
- "메시지 보내기"는 두 사용자 간 방을 생성하거나 기존 방을 반환(idempotent).
- 전송 경로: **STOMP WebSocket 우선**, 장애/난이도 시 **REST 폴링 폴백**(Q9).

**수용 기준**
- [ ] 보낸 메시지가 DB에 저장되고 재조회 시 동일하게 보인다.
- [ ] 같은 상대와 "메시지 보내기"를 반복해도 대화방은 하나로 유지된다.
- [ ] WebSocket 미사용 환경에서도 폴링으로 송수신이 가능하다.

## 4. 중고거래 (Market)

거래 유형(`tradeType`)에 따라 **노출/입력 필드가 다르다**: **사요(BUY)** / **팔아요(SELL)**.

### 4.1 필터
- 카테고리: 전체 / 먹거리(`FOOD`) / 장난감(`TOY`) / 생활용품(`DAILY`) / 의류(`CLOTHING`) / 기타(`ETC`).
  ('전체'는 미필터. Q1: 장난감 중복은 단일 `TOY`로 정리.)
- 거래완료 여부(`status`: `ON_SALE`/`SOLD`) 체크 필터.
- 거래 유형(`tradeType`: `BUY`/`SELL`) 필터.

### 4.2 목록 조회 (유형별 필드 차이)
- **팔아요(SELL)**: 제목, 가격, 이미지, **하트(좋아요) 개수**, 업로드 일자.
- **사요(BUY)**: 제목, 상세 내용, 작성자.

### 4.3 상세 조회
- **팔아요(SELL)**: 제목, 가격, 이미지, 하트 개수, 업로드 일자, **작성자**, 상세 내용.
- **사요(BUY)**: **상세 조회 없음** (Q2).

### 4.4 글쓰기
- 진입 시 **사요/팔아요 먼저 선택**.
- **사요(BUY)**: 제목, 상세 내용.
- **팔아요(SELL)**: 제목, 가격, 이미지, 상세 내용.
- **(LLM 연동)** 팔아요 글쓰기에는 **자동 작성 보조**(LLM-1)가 붙는다 → 6장 참조.

### 4.5 하트(좋아요)
- 팔아요 글에 **하트 토글** + 개수 집계.

**수용 기준**
- [ ] 목록/상세 응답 필드가 `tradeType`에 따라 위 명세대로 달라진다.
- [ ] BUY 글은 상세 조회 엔드포인트에서 제공되지 않는다.
- [ ] 하트는 사용자당 1회 토글되며 개수가 정확히 집계된다.
- [ ] `suggestedCategory`/입력 category는 `MarketCategory` enum 중 하나로 강제된다.

## 5. 저탄소 마켓플레이스 (Marketplace)

운영자/시드 데이터로 큐레이션된 저탄소 애견용품 마켓 (사용자 등록 없음, Q3).

### 5.1 필터 / 목록
- 카테고리 필터: 전체 / 먹거리(`FOOD`) / 생활용품(`DAILY`) / 기타(`ETC`).
- 목록 항목: **회사, 제목, 이미지, 별점(개수), 가격**, 그리고 **저탄소 한 줄 요약**(`lowCarbonSummary`).

### 5.2 상세 / 구매
- MVP는 **목록 조회까지만**. 상세/구매 플로우는 **추후**(Q3).

### 5.3 LLM 연동
- **(a)** 상품별 "왜 저탄소인지" 한 줄 요약: **시드 적재 시 1회 생성 → DB 캐싱**(매 요청 호출 금지).
- **(b)** 추천: 사용자 질의 + 본인 강아지 프로필 + 카탈로그를 grounding으로 주입 → 추천 목록 + 이유. (LLM-2 → 6장)

**수용 기준**
- [ ] 목록 응답에 시드값 별점/개수와 캐싱된 `lowCarbonSummary`가 포함된다.
- [ ] 추천 호출 시 본인 컨텍스트(강아지 프로필)가 자동 주입된다.
- [ ] 별점/리뷰 **작성** 기능은 제공하지 않는다(범위 밖).

## 6. LLM 기능 (요약 — 상세는 [llm-spec](./llm-spec.md))

| ID | 기능 | 엔드포인트 | 핵심 |
| --- | --- | --- | --- |
| **LLM-1** | 팔아요 판매글 자동 작성 | `POST /api/market/items/draft` | 사진(+키워드) → `{title,description,suggestedPrice,suggestedCategory}` 초안. 멀티모달(Nova Lite). 추천가=참고용. 결과는 초안 → 사용자 수정 후 작성 API 제출. |
| **LLM-2** | 저탄소 추천/코멘트 | (a) 시드 캐싱, (b) `POST /api/marketplace/recommendations` | (a) 한 줄 요약 1회 생성·캐싱, (b) query+강아지 프로필+카탈로그 grounding → 추천+이유. |

**공통**: 모델 `us.amazon.nova-lite-v1:0`(us-east-1), Converse API, **IAM Role only(Access Key 금지)**.
출력 JSON 스키마 강제 + 안전 파싱, 타임아웃, 실패 시 **폴백**(빈 초안/빈 추천)으로 본기능 비차단.

## 7. Out of Scope (이번 백엔드 범위 외)
- **커뮤니티**(게시판/피드): 프론트 화면만 구성, 백엔드 작업 없음.
- 저탄소 마켓 **상세/구매/리뷰 작성**.
- 소셜 로그인, 결제, 푸시 알림(SNS 연동은 인프라 옵션으로만 언급).
- 강아지 다중 등록(1:N) — 설계 여지만 둠.
